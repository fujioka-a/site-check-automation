# ピース実行用指示書

## タスク
商用公開サイト向けの目視チェック省力化ツールを、`Codex + Skill` ベースで実現するための要件整理・設計・実装を行ってください。  
目的は、インフラ改修や軽微なサイト修正の前後で、サイトに影響が出ていないことをエージェント経由で確認できるようにすることです。

## 背景
現在は DevTo を目視して確認しているが、省力化したい。  
速度計測や品質ゲートは初期スコープ外とし、まずは「見た目に影響する不具合」や「AWS移行に関連する不具合」を漏れなく把握しやすい仕組みを優先する。

## 参照資料
- 既存の `README`、`AGENTS`、関連ドキュメント
- 特に AGENTS に記載された以下の方針を遵守すること
  - Webサイトチェックは `site-check-analyzer` を使用
  - Playwright MCP を優先使用
  - 同一ドメイン制限を守る
  - 購入 / 削除 / フォーム送信 / ログアウト / データ更新は禁止
  - 見た目に影響するエラーを最優先
  - Console ノイズは過剰に重要視しない
  - AWS移行に関連するエラーを優先検出
  - 出力は Markdown で構造化し、重大度別に整理し、原因推定を必ず記載

## 実現したい要件
- 対象はログイン不要の公開ページのみ
- 商用環境を対象とする
- 実行は `Codex` などのエージェント経由を前提とする
- 入力URLを起点に、同一ドメイン内を自動クロールしてチェックする
- 可能な限り網羅する。ただしクロール上限は最大100ページ、最大深さ4とする
- 一覧ページなどで同列のリンクが大量に並ぶ場合は、代表サンプルとして先頭10件程度を優先的にチェック対象とする
- 一覧ページやクロール境界の都合で未チェック領域が残る場合は、レポートに未確認範囲を分かりやすく明記する
- チェック方式は両対応にする
  - 単体サイトのチェック・評価
  - 同ツールが出力した改修前後2本の結果レポートの差分比較
- 各ページごとに Console の `warning` / `error` を収集する
- 差分比較では、Console 差分の有無と内容を確認でき、想定外の差分を判別しやすいようにする
- 出力には以下を含める
  - 重大度別の指摘
  - 根拠
  - 原因推定
  - スクリーンショット
  - 比較差分
  - 総合判定
- 総合判定は `問題なし / 要確認 / 問題あり` の3段階とする
- 将来的に 1000 超のサイトでも、事前管理を最小化してオンデマンドに URL を渡して実行できる方向を意識する

## 判定方針
- `error` は原則として NG 寄りに扱う
- `warning` は差分増加時に要確認として扱う
- 見た目に影響する異常を最優先で報告する
- AWS移行に関連すると推定される異常は優先的に拾う
- 許容リストは初期スコープに含めない
- Console の既知ノイズは、差分比較や総合判定で過大評価しない。よくある Warning は補足としてレポートに記載してよい

## 制約
- 後方互換コードは不要。明示的に必要と判断されない限り入れない
- 禁止操作は実行しないこと
  - 購入
  - 削除
  - フォーム送信
  - ログアウト
  - データ更新
- 同一ドメイン制限を守ること
- 初期スコープでは速度計測、品質ゲート、ログイン後画面対応、許容リスト管理は含めない

## ピースへの期待
1. 既存の README / AGENTS / 関連資料を読み、現状の方針と制約を整理する
2. 既存実装があれば流用可能箇所を調査する
3. この要件を満たすための最小構成を設計する
4. 単体チェックと前後比較の両方を扱える実装を行う
5. クロール範囲、大量一覧の絞り込みルール、未確認範囲の扱いを明示できるレポート設計にする
6. `Codex` などのエージェント経由で実行しやすいインターフェースを整える
7. テスト、型チェック、必要な検証を行う

## 完了条件
- エージェント経由で URL を渡して実行できる
- 同一ドメイン内をクロールしてチェックできる
- 単体チェック結果を Markdown で構造化出力できる
- 前後2本の結果レポートを比較した結果を Markdown で構造化出力できる
- 各ページの Console `warning` / `error` を収集できる
- 未確認範囲がある場合にレポートへ明記できる
- 総合判定 `問題なし / 要確認 / 問題あり` を出せる
- 禁止操作を避ける実装になっている
- ビルド / テストが通る
